# Windows 版本技术选型分析（AI 辅助开发视角）

> 在计划使用 AI（如 Cursor、GitHub Copilot、Claude 等）辅助实现的前提下，从「AI 生成质量」「可维护性」「调试难度」等维度分析技术选型。

---

## 一、AI 辅助开发的核心考量

### 1.1 AI 擅长的领域

| 维度 | 说明 | 对选型的影响 |
|------|------|--------------|
| **训练数据丰富度** | 常见技术栈在训练数据中占比高 | 优先选择主流、成熟技术 |
| **模式标准化** | 重复性高的样板代码 | P/Invoke、MVVM、配置读写等 AI 表现好 |
| **文档完善** | 官方文档、Stack Overflow 多 | 减少 AI 幻觉和过时信息 |
| **项目可参考** | 有类似开源项目 | 可让 AI 参考 PowerToys、Rectangle 源码 |

### 1.2 AI 的薄弱环节

| 维度 | 说明 | 对选型的影响 |
|------|------|--------------|
| **复杂状态与异步** | 多线程、回调、事件顺序 | 避免过度依赖复杂并发模型 |
| **平台细节** | Win32 标志位、结构体布局、错误码 | 需要人工校验 P/Invoke 和结构体 |
| **冷门 API** | 使用少、文档少的 API | 尽量用常见 API |
| **构建与工具链** | 复杂构建、多项目依赖 | 保持项目结构简单 |

### 1.3 结论导向

- **语言**：C# 优于 C++（训练数据多、模式清晰、错误更易定位）
- **UI**：WPF 优于 WinUI 3（生态成熟、示例多）
- **架构**：清晰分层、小步迭代，便于 AI 分模块生成和修改

## 二、语言选型：C# vs C++

### 2.1 C#（推荐）

| 维度 | 评分 | 说明 |
|------|------|------|
| AI 生成质量 | ⭐⭐⭐⭐⭐ | 训练数据量大，语法和模式稳定 |
| 样板代码 | ⭐⭐⭐⭐⭐ | 属性、LINQ、async/await 等 AI 生成准确 |
| 错误定位 | ⭐⭐⭐⭐⭐ | 编译错误明确，异常堆栈清晰 |
| 调试体验 | ⭐⭐⭐⭐⭐ | Visual Studio 支持好，AI 能理解断点、日志 |
| 参考项目 | ⭐⭐⭐⭐ | 大量 C# 窗口管理、托盘、热键示例 |

**AI 友好点：**

- P/Invoke 声明：网上示例多，AI 能正确生成 `[DllImport]`、`MarshalAs` 等
- 结构体：`[StructLayout(LayoutKind.Sequential)]` 等模式固定
- 配置：`System.Text.Json` 序列化/反序列化代码 AI 生成稳定
- 异步：`Task`、`async/await` 模式统一，AI 不易出错

**需人工把关：**

- P/Invoke 的 `CharSet`、`ExactSpelling`、`SetLastError` 等细节
- `IntPtr` 与 `HWND` 的转换、空指针判断
- 多显示器、DPI 等平台相关逻辑

### 2.2 C++（不推荐用于 AI 主导开发）

| 维度 | 评分 | 说明 |
|------|------|------|
| AI 生成质量 | ⭐⭐⭐ | 易生成有内存泄漏、未初始化等问题的代码 |
| 样板代码 | ⭐⭐ | 头文件、预处理器、构建配置复杂 |
| 错误定位 | ⭐⭐ | 模板、链接错误对 AI 理解困难 |
| 调试体验 | ⭐⭐⭐ | 需理解内存、指针，AI 辅助有限 |
| 参考项目 | ⭐⭐⭐⭐ | PowerToys 可参考，但架构复杂 |

**主要风险：**

- 智能指针、RAII、移动语义：AI 容易出错，需人工审查
- 头文件与实现分离：AI 容易生成不一致的声明
- CMake/msbuild：构建脚本复杂，AI 修改容易破坏配置
- 与 PowerToys 同技术栈，但 PowerToys 是大型团队维护，单人 + AI 难以驾驭

**结论**：除非有强 C++ 背景或追求极致性能/零依赖，否则不推荐用 C++ 做 AI 主导开发。

## 三、UI 框架：WPF vs WinUI 3

### 3.1 WPF（推荐）

| 维度 | 评分 | 说明 |
|------|------|------|
| AI 训练数据 | ⭐⭐⭐⭐⭐ | 存在时间长，示例多 |
| XAML 生成 | ⭐⭐⭐⭐ | 布局、绑定、样式常见模式 AI 能生成 |
| 托盘/系统集成 | ⭐⭐⭐⭐⭐ | `NotifyIcon`、`System.Windows.Forms` 等成熟 |
| 学习曲线 | ⭐⭐⭐⭐ | 资料多，AI 能基于文档生成合理代码 |

**AI 友好点：**

- 数据绑定：`INotifyPropertyChanged`、`ObservableCollection` 模式固定
- 命令：`ICommand`、`RelayCommand` 等实现常见
- 设置窗口：列表、滑块、开关等控件 AI 能生成
- 快捷键录制：`KeyEventArgs`、`ModifierKeys` 等处理简单

**注意点：**

- XAML 较冗长，可让 AI 先生成结构，再人工精简
- 样式、主题：尽量用系统自带或简单样式，减少自定义

### 3.2 WinUI 3（已选定）

| 维度 | 评分 | 说明 |
|------|------|------|
| AI 训练数据 | ⭐⭐⭐ | 较新，示例少，需更多人工校验 |
| 现代化程度 | ⭐⭐⭐⭐⭐ | Fluent Design、新控件 |
| 系统要求 | ⭐⭐⭐⭐ | Windows 10 1809+ |
| 生态成熟度 | ⭐⭐⭐ | 第三方库、文档不如 WPF |

**选定后的适配建议：**

- **系统托盘**：使用 H.NotifyIcon（NuGet），AI 可参考其文档生成集成代码
- **全局快捷键**：参考 [whid.eu 教程](https://whid.eu/2022/05/13/chapter-5-add-global-hot-key-in-winui-3/)，将链接提供给 AI 作为上下文
- **CsWin32**：用 `NativeMethods.txt` 声明所需 API，由源生成器生成，减少 P/Invoke 手写错误

## 四、关键技术模块的 AI 适配度

### 4.1 高适配度（可优先交给 AI）

| 模块 | 原因 | 建议 |
|------|------|------|
| 配置读写 (JSON) | 模式固定，示例多 | 让 AI 生成 Config + 序列化代码 |
| 布局计算 | 纯数学逻辑，无平台依赖 | 参考 Rectangle 的 Calculation 类，AI 可移植 |
| 窗口操作封装 | P/Invoke 声明有大量示例 | 提供 Win32 文档链接，让 AI 生成后人工校验 |
| 快捷键映射 | 字典/枚举映射 | 直接让 AI 生成 |
| 设置界面 | 标准 WPF 控件 | 描述需求，AI 生成 XAML + ViewModel |

### 4.2 中等适配度（需人工校验）

| 模块 | 原因 | 建议 |
|------|------|------|
| 全局快捷键 | 需消息窗口、WM_HOTKEY 处理 | AI 生成骨架，人工校验窗口句柄、注册逻辑 |
| 多显示器 | 工作区、DPI 等细节多 | 分步实现，每步用 AI 生成后人工测试 |
| 拖拽检测 | 钩子、坐标转换 | 先实现简化版，再让 AI 协助扩展 |

### 4.3 低适配度（建议人工主导）

| 模块 | 原因 | 建议 |
|------|------|------|
| 足迹预览窗口 | 透明、置顶、无边框等 | 人工实现核心逻辑，AI 协助样式 |
| 安装/打包 | MSIX、签名等 | 参考官方文档，人工配置 |
| 权限/管理员 | 清单、提升逻辑 | 人工查阅文档后实现 |

## 五、选定技术栈：C# + WinUI 3（AI 辅助适配）

| 层级 | 选择 | 理由 |
|------|------|------|
| 语言 | C# | 训练数据多、模式清晰、错误易定位 |
| 运行时 | .NET 8 | 长期支持、AI 对 .NET 支持好 |
| UI | **WinUI 3** | 已选定，Fluent Design、现代化 |
| 配置 | System.Text.Json | 内置、API 简单、AI 生成稳定 |
| JSON 配置路径 | 文件 (`%APPDATA%`) | 简单、跨平台思维一致 |
| Win32 封装 | **CsWin32** | 源生成器减少手写错误，推荐 |
| 系统托盘 | H.NotifyIcon | WinUI 3 无内置，此库成熟且 MIT |

### WinUI 3 下的 AI 辅助策略

| 模块 | 策略 |
|------|------|
| 托盘 | 提供 H.NotifyIcon 文档/示例给 AI，生成集成代码 |
| 快捷键 | 提供 whid.eu 教程链接，让 AI 生成消息钩子逻辑 |
| P/Invoke | 使用 CsWin32，在 NativeMethods.txt 中列出 API 名 |
| 设置页 | 用 WinUI 3 的 NavigationView、ToggleSwitch 等，描述需求让 AI 生成 |

### 不推荐或延后引入

| 技术 | 原因 |
|------|------|
| 复杂 MVVM 框架 | 增加样板代码，AI 易出错 |
| 手写 P/Invoke | 易出错，优先用 CsWin32 |
| C++ | 对 AI 主导开发不友好 |

## 六、AI 辅助开发工作流建议

### 6.1 任务拆分粒度

| 粒度 | 示例 | 适合 AI |
|------|------|----------|
| 过小 | 「写一个 P/Invoke」 | ✅ 容易 |
| 合适 | 「实现 LeftHalfCalculator，输入屏幕工作区，输出左半屏 RECT」 | ✅ 推荐 |
| 过大 | 「实现整个窗口管理模块」 | ❌ 易遗漏、易出错 |

### 6.2 提示词策略

- **提供上下文**：附上 Rectangle 源码片段或文档链接
- **明确输入输出**：如「输入：MonitorInfo 结构体；输出：C# RECT」
- **分步请求**：先要结构体定义，再要计算逻辑，最后要集成
- **指定风格**：如「使用 C# 10 语法」「遵循 async/await 模式」

### 6.3 人工校验清单

- [ ] P/Invoke 的 `DllImport` 属性、参数类型
- [ ] 结构体 `[StructLayout]`、字段顺序、字节对齐
- [ ] 句柄、指针的空值检查
- [ ] 多显示器、DPI 下的坐标计算
- [ ] 全局钩子、热键的注册与注销时机

## 七、总结

| 决策 | 选定 | 理由 |
|------|------|------|
| 语言 | **C#** | 训练数据多、模式清晰、错误易定位 |
| UI | **WinUI 3** | 已选定，Fluent Design、现代化 |
| 运行时 | **.NET 8** | LTS、支持好 |
| Win32 | **CsWin32** | 源生成器减少 P/Invoke 错误 |
| 托盘 | **H.NotifyIcon** | WinUI 3 无内置，此库成熟 |
| 架构 | **清晰分层 + 小步迭代** | 便于 AI 分模块生成和修改 |

在 C# + WinUI 3 下，通过 **CsWin32**、**H.NotifyIcon** 和参考教程（whid.eu 等）作为 AI 上下文，可弥补 WinUI 3 生态较新的不足，实现高效 AI 辅助开发。
